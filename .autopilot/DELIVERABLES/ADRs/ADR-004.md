# ADR-004: Font bundling approach

Status: Accepted
Date: 2026-05-08

## Context
The design system requires Newsreader (serif, Google Fonts) and Mulish (sans, Google Fonts) for all text rendering. iOS does not include these fonts. Options: bundle TTF files, use SF Pro New York as a fallback, or download at runtime.

## Decision
Bundle the TTF font files in the app and register them in `Info.plist` under `UIAppFonts`. Implement `Font.custom("Newsreader-Regular", size:)` wrappers in the design system. If fonts are absent at build time, the system falls back to San Francisco (which degrades gracefully in previews).

## Rationale
- The brief states "design system pixel-perfect to the handoff spec" — substituting system fonts deviates from this.
- The handoff README explicitly says "Type-rendering matters. The serif (Newsreader) sets the literary tone — don't substitute with a generic system serif unless you really must."
- Runtime download of fonts would introduce a network call, violating NFR-001 (local-only, no network).
- Bundling adds ~400KB per weight variant. For 7 variants (Regular, Italic, Medium + Mulish Regular, Medium, SemiBold, Bold) the total is ~2.5MB — acceptable for an iOS app.
- Font files must be obtained from Google Fonts (open license, SIL OFL) and dragged into Xcode's project navigator before final build. The code uses correct PostScript names.

## Consequences
- `Info.plist` must include `UIAppFonts` array with all 7 font file names.
- Developer must obtain font TTF files and add them to the Xcode target before submission.
- Font names must match the exact PostScript name (e.g. "Newsreader-Regular", "Mulish-Bold").
- In CI/CD environments without the font files, UI tests will render system font — visually incorrect but functionally passing.
