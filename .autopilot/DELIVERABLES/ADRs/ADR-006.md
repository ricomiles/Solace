# ADR-006: Single AppState class for navigation state

Status: Accepted
Date: 2026-05-08

## Context
Writing flows span multiple views: mood check-in → writing view. These views need to share the selected mood and the writing mode. Options: pass state via view initializer arguments, or use a shared observable state object.

## Decision
Use a single `@Observable AppState` class injected at the root, containing all navigation flags and writing session state.

## Rationale
- The writing flow involves 3+ screens in sequence (Home → MoodCheckIn → WritingView). Passing state through `.fullScreenCover` bindings across this chain would require threading bindings through intermediate views.
- A shared `AppState` avoids prop-drilling and keeps the navigation logic in one place.
- `AppState` is not a view model — it holds no domain data (entries, prompts). It only holds navigation state and the in-flight writing session.
- `@Observable` is used (see ADR-002) so only the properties actually accessed by a view trigger re-renders.

## Consequences
- `AppState` must be injected at `SolaceApp` level and read via `@Environment(AppState.self)` in views that need it.
- Writing session state (`selectedMood`, `pendingWritingMode`) is reset by `AppState.resetWritingSession()` when a writing view is dismissed.
- `AppState.moodLoggedToday` is a computed Bool derived from querying today's entries for any non-nil mood; it is not persisted.
